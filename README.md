## 离线构建阶段

#### 1. 背景与问题定义

现实中的口腔问诊并不是完全随机的。医生通常会围绕一些高价值问题逐步缩小范围

例如：

- 是否遇冷、遇热刺激疼痛；
- 是否有自发痛、夜间痛、放散痛；
- 是否有牙龈红肿、溢脓、松动；
- 是否拔过牙、补过牙、戴过修复体；
- 疼痛位置是否在智齿、门牙、后牙或牙龈区域。

这些问题本质上构成一棵“问诊决策树”：每个节点提出一个能有效切分候选病例集合的问题，用户回答“是 / 否 / 不确定”后，系统进入下一层候选空间。

早期方案可以把这类问题理解为“热词策略”：预先找到高频、高区分度的医学特征词，并在对话中验证它们是否成立。但仅靠人工热词存在两个问题：

1. **特征顺序难以人工维护**：前面的问题会影响后续问题。例如用户已经确认“智齿区域疼痛”，系统就不应再追问“是不是门牙疼”。
2. **热词无法覆盖所有边界情况**：当候选集已经很小，或病例之间高度相似时，通用热词可能无法继续有效切分候选集合。

因此，当前系统采用“离线构建特征空间 + 离线构建决策树 + 在线动态补充 probe”的架构。离线阶段的目标是：

- 将病例文本转化为数值化特征；
- 自动发现高区分度的问诊维度；
- 构建一棵可解释的 yes/no 决策树；
- 为在线问诊提前生成自然语言追问。

#### 2. 放弃字符串匹配而使用语义匹配

用户输入通常是口语化的，而病例描述通常是书面化、医学化的。二者之间存在明显表达鸿沟：

| 用户表达 | 病例描述 | 字符串匹配结果 | 语义匹配结果 |
|---|---|---:|---:|
| “喝凉水牙酸” | “冷刺激引起短暂疼痛” | 很弱 | 强 |
| “晚上自己疼醒” | “夜间自发性疼痛” | 很弱 | 强 |
| “牙龈鼓包流脓” | “根尖区瘘管，可见脓性渗出” | 弱 | 中高 |

字符串匹配依赖字面重叠，无法稳定处理同义表达、口语表达、否定表达和中英文混杂表达。当前系统改为使用 LLM embedding 进行语义向量化，将病例文本、句子窗口、关键词概念统一映射到向量空间中，再通过 cosine 相似度衡量语义接近程度。

离线阶段主要由两个脚本完成：

```text
resources/raw/cases_demo.json
        │
        ▼
scripts/build_vectors.py
        │
        ├── resources/case_vectors.json
        └── resources/feature_clusters.json
        │
        ▼
scripts/build_tree.py
        │
        └── resources/case_question_tree.json
```

其中：

- `build_vectors.py` 负责构建病例向量、语义簇、概念簇和病例特征矩阵；
- `build_tree.py` 读取预计算特征矩阵，在数值空间中构建问诊决策树；
- 之后在线问诊只需要加载这些离线结果，不需要重新扫描全部病例文本。

#### 4. 构建病例向量，并分簇：build_vectors.py

离线向量构建分为三条互补路径：

1. **文档级 / 病例级向量**：用于在线首轮粗排；
2. **句子窗口级向量**：用于发现病例中的局部症状模式；
3. **关键词概念级向量**：用于精细区分相似病例。

#### 4.1文档级向量

对每条病例，将 `title + description + treatment` 拼接成完整文本，并计算一个病例级向量：

```text
case_text = title + "\n" + description + "\n" + treatment
sentence_vec = embed(case_text)
```

这个向量主要用于在线首轮检索。例如用户输入“智齿那里晚上疼得厉害”，系统会计算用户输入的向量，并与每条病例的 `sentence_vec` 求 cosine 相似度，用来快速找到大方向相关的病例。

病例级向量的优点是召回稳定、速度快；缺点是粒度较粗，不适合区分“冷刺激短暂疼痛”和“热刺激加重疼痛”这类细节差异。

#### 4.2句子级向量

仅使用整段病例 embedding 会损失局部信息。为保留症状片段，系统将每个病例描述切成多粒度窗口：

```text
原始描述：
  夜间自发性剧烈跳痛，冷水可暂时缓解，热刺激使疼痛加重，疼痛不能定位到具体牙齿。

窗口示例：
  夜间自发性剧烈跳痛
  冷水可暂时缓解
  热刺激使疼痛加重
  夜间自发性剧烈跳痛，冷水可暂时缓解
  冷水可暂时缓解，热刺激使疼痛加重
```

每个窗口都计算 embedding，然后将相似窗口聚类。聚类后，每个簇代表一种局部症状模式。

例如：

- 夜间自发性疼痛；
- 冷热刺激敏感；
- 牙龈红肿出血；
- 咬合痛；
- 溢脓或瘘管；
- 修复体松动。

对每个语义簇，计算簇内向量的平均向量作为该簇的中心向量 `centroid`。

#### 4.3词语级向量

句子窗口能保留局部症状，但有时仍不够细。例如：

```text
“冷热刺激可有轻中度敏感”
“冷刺激引起短暂疼痛”
“热刺激可能使疼痛加重”
```

这些句子在 embedding 空间中可能很接近，但临床含义并不完全相同。为进一步区分细粒度特征，系统引入 LLM 自动概念抽取。

对每条病例，LLM 提取若干有区分意义的概念：

```json
{
  "concepts": [
    {"term": "自发痛", "role": "疼痛是否需要外界刺激触发", "importance": "high", "negative": false},
    {"term": "夜间加重", "role": "疼痛发作的时间规律", "importance": "high", "negative": false},
    {"term": "跳痛", "role": "疼痛的性质和节律感", "importance": "medium", "negative": false},
    {"term": "冷水缓解", "role": "温度刺激对疼痛的影响方向", "importance": "high", "negative": false},
    {"term": "叩痛不明显", "role": "垂直方向机械刺激的敏感程度", "importance": "medium", "negative": true},
    {"term": "放散痛", "role": "疼痛是否能定位到具体牙齿", "importance": "medium", "negative": false}
  ]
}
```

这里有两个重要字段：

- `term`：病例中的关键词或症状概念；
- `role`：该概念在临床鉴别中的作用；
- `negative`：该特征是否是“明确不出现”的阴性特征。

`negative=true` 非常重要。例如“叩痛不明显”虽然表示不存在明显叩痛，但它本身就是区分牙髓炎与根尖周炎的重要依据。

#### 4.3.1 role聚类

不同病例中，LLM 对同类概念的 role 描述可能不同：

```text
疼痛是否需要外界刺激触发
疼痛的自发性程度
不依赖外界因素的疼痛发作
有无外界诱因即出现的痛感
```

这些表达都指向同一临床维度：**疼痛自发性**。

因此系统将所有 role 文本 embedding 后再次聚类，并让 LLM 为每个簇生成短标签：

```json
[
  {"id": 0, "name": "疼痛自发性", "examples": ["自发痛(疼痛是否需要外界刺激触发)"]},
  {"id": 1, "name": "温度刺激反应", "examples": ["冷水缓解(温度刺激对疼痛的影响方向)"]},
  {"id": 2, "name": "疼痛时间规律", "examples": ["夜间加重(疼痛发作的时间规律)"]},
  {"id": 3, "name": "叩诊敏感度", "examples": ["叩痛不明显(垂直方向机械刺激的敏感程度)"]}
]
```

这样可以在不预设固定标签体系的前提下，自动形成一组可解释的**关键词维度**。

#### 4.4 特征投影：将病例映射到固定维度

聚类完成后，系统得到两类特征簇：

- `K` 个句子窗口语义簇；
- `M` 个关键词概念簇。

对于每个病例，系统计算它与每个簇的相关程度，得到一个固定长度的特征向量：

```text
feature_vec(case_i) = [sem_1, sem_2, ..., sem_K, concept_1, concept_2, ..., concept_M]
```

```
case_042.feature_vec = [
    # K 个语义簇维度
    0.95,  # dim_0: 夜间自发性跳痛
    0.87,  # dim_1: 冷热刺激相关疼痛
    0.18,  # dim_2: 牙龈红肿出血
    0.82,  # dim_3: 咬合痛
    ...,

    # M 个概念簇维度
    0.91,  # 疼痛自发性
    0.88,  # 温度刺激反应
    0.93,  # 疼痛时间规律
    0.15,  # 叩诊敏感度
    0.10,  # 牙齿松动程度
]
```

这一步把非结构化病例文本转换成“每个临床维度上有多相关”的数值。之后构建决策树时，系统不再处理原始文本，只在这个固定维度空间中寻找最佳切分。

#### 4.4.1 语义簇匹配 

对病例 `case_i` 和语义簇 `k`：

```text
sem_k(case_i) = max cosine(embed(window), centroid_k)
                window ∈ windows(case_i)
```

即：只要病例中的某个窗口强烈匹配该语义簇，该维度就会得到高分。

#### 4.4.2 概念簇匹配

对病例 `case_i` 和概念簇 `m`：

```text
concept_m(case_i) = max cosine(embed(role), centroid_m)
                    role ∈ extracted_roles(case_i)
```

即：只要病例抽取出的某个概念 role 与该概念簇接近，该维度就会得到高分。

#### 5.决策树构建：build_tree.py

#### 5.1 构建原理

每个节点对应当前候选病例集合 `C`。系统需要在所有特征维度中选择一个最佳维度 `dim`，将 `C` 切成三部分：

- `positive_ids`：该特征明显成立；
- `negative_ids`：该特征明显不成立；
- `unknown_ids`：暂不确定。

```
dim_0 = [
    0.95,  # case_0: 干槽症
    0.87,  # case_1: 蛀牙
    0.85,  # case_2: 溃疡
    0.82,  # case_3: 牙龈出血
    ...,
    0.39,  # 
    0.20,  # 
    0.19,  # 
    0.15,  # 
    0.10,  # 
]
```



用户回答“是”时，候选集进入 positive 分支；回答“否”时，进入 negative 分支；回答“不确定”时，则保留当前信息，询问该节点的候选维度。

为了让决策树的问题更接近自然 yes/no 问题，当前切分不再随意寻找任意阈值，而是在语义边界附近寻找阈值。

核心函数是 `semantic_split`：

```python
anchor = 0.50
search_range = 0.12
margin = 0.03

lo = anchor - search_range  # 0.38
hi = anchor + search_range  # 0.62
```

系统只考虑 `[0.38, 0.62]` 附近的阈值。只有当某个维度上同时存在明显高于 `hi` 和明显低于 `lo` 的病例时，才认为这个维度具有切分价值。

对某个候选阈值 `θ`：

```text
positive: value >= θ + margin
unknown:  θ - margin <= value < θ + margin
negative: value < θ - margin
```

这样设计的好处是：

- 避免在全体值都很高或都很低的维度上强行切分；
- 让问题语义更稳定，例如“是否存在某种症状特征”；
- 将模糊边界保留为 unknown，降低误分风险。

#### 5.2 评分标准

对每个候选切分，系统计算 `split_quality`：

```text
split_quality = entropy × coverage × balance × unknown_penalty
```

其中：

- `entropy`：positive 与 negative 越接近 1:1 越高；
- `coverage`：能明确归入 positive/negative 的比例越高越好；
- `balance`：避免一边过大、一边过小；
- `unknown_penalty`：unknown 越多，分数越低。

这保证了系统优先选择既有临床意义、又能有效切分当前候选集的问题。

#### 5.3 设计思路：理想的运行逻辑

浅层节点面对大量病例，应该优先用粗粒度语义特征快速分流；深层节点面对少量相似病例，应该优先用关键词概念维度精细区分。

因此系统对不同深度使用不同权重：

```python
if dim < K:      # 句子窗口语义维度
    weight = max(0.3, 1.0 - depth * 0.12)
else:            # 关键词概念维度
    weight = min(1.5, 0.8 + depth * 0.12)
```

最终节点选择依据为：

```text
adjusted_score = split_quality × depth_weight
```

这使得：

- 根节点更倾向使用大范围症状模式；
- 中后层更倾向使用“疼痛诱因”“病程特征”“鉴别阴性指征”等细粒度概念。

#### 5.4 LLM辅助组织语言

树节点选出某个维度后，该维度本身通常是技术性标签，例如：

```text
疼痛症状状态
温度刺激反应
鉴别阴性指征
牙体评估因素
```

直接把这些标签问给用户并不自然。因此系统会将标签和代表性证据交给 LLM 改写为口语化问题：

```text
技术标签：温度刺激反应
代表证据：冷水刺激后短暂疼痛；热刺激使疼痛加重

生成问题：
  “你的牙碰到冷的或热的东西时，会不会明显疼起来？”
```

这样，离线树中不仅保存了切分所需的病例 ID，也保存了可直接用于在线问诊的问题文本。

#### 6. 观察生成的决策树

当前构建结果显示，树的每个节点都由一个特征维度控制。例如根节点为：

```text
n [dim=202(con) score=0.4461 cases=520 label=病变位置]
  yes→ n_y [dim=201(con) score=0.549 cases=162 label=牙体评估因素]
  no → n_n [dim=206(con) score=0.3763 cases=358 label=病变定位]
```

这说明当前树在根部优先选择了概念维度，而不是句子窗口语义维度。后续节点中也频繁出现：

- 症状与体征；
- 患者主诉；
- 症状诱因；
- 疼痛症状状态；
- 病变范围；
- 龋损特征描述；
- 温度刺激反应；
- 鉴别阴性指征；
- 影像学征象。

从当前树也能看到一个需要继续优化的问题：部分概念标签仍偏泛化，例如“患者主诉”“主观描述”“症状与体征”。这类标签虽然能切分数据，但作为问诊问题的可解释性不够强。后续可以通过改进 LLM 概念抽取 prompt、过滤过泛标签、或提高具体临床 role 的权重来改善。



## 在线交互阶段

#### 1. 流程

```text
用户自然语言输入
   ↓
转成语义查询向量 + 关键词向量
   ↓
在当前候选病例集中打分排序
   ↓
根据决策树或动态 probe 选择下一问
   ↓
解析用户回答
   ↓
继续缩小候选集，直到输出最可能病例
```

因此，在线阶段本质上是一个“向量检索 + 交互式决策树 + 候选集重排序”的循环系统。

在线流程可以概括为五个节点：

```text
parse_input
   ↓
narrow
   ↓
score
   ↓
route
   ├── ask
   └── answer
```

对应到问诊逻辑：

```text
1. parse_input：理解用户输入
2. narrow：根据已有回答缩小候选集
3. score：对候选病例重新排序
4. route：判断继续追问还是输出结果
5. ask / answer：生成下一问或给出最终匹配病例
```

#### 2. 理解用户输入：parse_input.py

判断用户当前输入是一个“新问题”，还是在回答上一轮 probe。

#### 2.1 新问题

例如用户第一次说：

```text
“右边最里面的牙疼，晚上更明显，吃冷的会舒服一点。”
```

系统调用 LLM 抽取结构化信息：

```json
{
  "query_cn": "右侧后牙夜间自发痛，冷刺激缓解",
  "query_en": "right posterior tooth spontaneous night pain relieved by cold stimulus",
  "intent": "symptom",
  "concepts": [
    {
      "term": "右侧后牙",
      "role": "疼痛发生的位置"
    },
    {
      "term": "夜间痛",
      "role": "疼痛发作的时间规律"
    },
    {
      "term": "冷刺激缓解",
      "role": "温度刺激对疼痛的影响方向"
    }
  ]
}
```

随后系统做两类 embedding：

```text
query_sentence_vec = embed(query_cn)

query_keyword_vecs = [
  embed("右侧后牙"),
  embed("夜间痛"),
  embed("冷刺激缓解")
]
```

其中：

```text
query_sentence_vec
  用于整体语义粗排

query_keyword_vecs
  用于关键词级精细匹配
```

这一步只在新问题开始时执行。后续用户回答“是的”“不是”“不确定”时，系统不会把这些短回答重新当作新的主查询，否则会污染原始症状向量。

#### 2.2 做回答

如果上一轮系统问：

```text
“你的牙齿会不会晚上自己疼起来，不碰也疼？”
```

用户回答：

```text
“会，晚上尤其明显。”
```

系统调用 LLM 解析该回答：

```json
{
  "signal": "yes",
  "confidence": 0.92,
  "evidence": "晚上尤其明显",
  "new_observations": ["夜间痛明显"]
}
```

其中 `signal` 只能是：

```text
yes
no
uncertain
unrelated
```

解析结果会用于更新 memory：

```text
yes       → confirmed[probe_id]
no        → denied[probe_id]
uncertain → uncertain[probe_id]
unrelated → 不更新 probe 状态
```

如果回答中带有新的症状信息，例如“晚上尤其明显”，系统会把它追加为新的 keyword 向量，增强后续排序。

在线阶段的核心是 `Memory`。它保存整个问诊过程中的状态

#### 3. 决策树分支选择：narrow.py

`narrow` 节点负责根据当前状态缩小候选病例集合。

如果用户已经回答了某个树节点问题，系统会进入对应子节点：

```text
用户回答 yes → tree_node = yes_child
用户回答 no  → tree_node = no_child
```

每个树节点本身保存一批病例 ID：

```json
{
  "id": "n_y_n",
  "case_ids": ["case_042", "case_057", "case_081", "..."]
}
```

所以只要进入该节点，候选集就可以先缩小到这个节点覆盖的病例。

#### 4. 候选病例评分：score.py

```text
score =
  α(N) × sentence_sim
+ β(N) × keyword_sim
+ γ(N) × probe_score
if N > 50:
    α, β, γ = 0.55, 0.20, 0.25
elif N > 20:
    α, β, γ = 0.40, 0.30, 0.30
elif N > 5:
    α, β, γ = 0.25, 0.35, 0.40
else:
    α, β, γ = 0.10, 0.35, 0.55
候选集大时：
  主要依赖 sentence_sim 做方向性粗排。

候选集变小时：
  逐步提高 keyword_sim 和 probe_score 的权重。

候选集很小时：
  主要依赖用户回答过的 probe 进行精排。
```

其中 N 是当前候选集大小。

#### 4.1 句子级相似度 sentence_sim

```python
sentence_sim = cosine(query_sentence_vec, case_sentence_vec)
```

它衡量用户整体描述与病例全文的语义相似度。

例如：

```text
用户：右边后牙晚上自己疼，喝冷水舒服点
病例：急性牙髓炎，夜间自发痛，冷刺激可暂时缓解
```

这两段文本即使没有完全相同的词，也会因为语义接近而得到较高 cosine 分数。

`sentence_sim` 的作用主要是粗筛。候选集很大时，它帮助系统快速找到大致方向。

#### 4.2 关键词级相似度 keyword_sim

关键词级匹配不是字符串匹配，而是向量匹配。

用户输入被 LLM 抽成若干关键词：

```text
["右侧后牙", "夜间痛", "冷刺激缓解"]
```

病例离线阶段也抽取了关键词，并分成：

```text
positive keyword vectors
negative keyword vectors
```

在线评分时，对用户每个关键词向量，在病例关键词向量中找最佳匹配：

```python
positive_score =
    avg_best_match(query_keyword_vecs, case.positive_keyword_vecs)

negative_score =
    avg_best_match(query_keyword_vecs, case.negative_keyword_vecs)

keyword_sim =
    positive_score - 0.4 × negative_score
```

这里 negative keyword 的意义很重要。

例如某病例明确描述：

```text
“无叩痛”
```

那么“叩痛不明显”会作为 negative 特征存储。如果用户明确说“敲牙很痛”，该病例就应当被惩罚，而不是仅仅因为都出现了“叩痛”相关语义就被奖励。

#### 4.3 probe_score

`probe_score` 来自用户对系统追问的回答。

如果某病例属于已确认 probe 的 positive 集合，得到奖励：

```text
confirmed + positive → 加分
```

如果某病例属于已确认 probe 的 negative 集合，受到惩罚：

```text
confirmed + negative → 减分
```

probe_score 体现的是交互式问诊的价值。越到后期，用户已经回答的问题越多，probe_score 越能区分相近病例。

#### 4.4 结束判断

停止条件包括：

```text
1. 没有候选病例；
2. 只剩 1 个候选病例；
3. 已达到最大问诊轮数；
4. top1 分数足够高，且 top1 与 top2 拉开明显差距；
5. 已经找不到有效 probe 继续切分候选集。
```

#### 5. 追问生成：ask.py

在线追问有三层策略。

```text
第一层：使用离线决策树 probe
第二层：动态挖掘当前候选集 probe
第三层：逐例确认
```

#### 5.1 第一层：离线决策树 probe

系统优先使用离线阶段构建好的决策树。

当前候选集会被映射到一个树节点，然后从该节点的 probes 中选择最合适的问题。

速度快，因为问题已经离线生成，在线只需要读取。

#### 5.2 第二层：动态 probe

如果当前树节点已经是叶子，或者树上的 probe 已经问过，系统会启用动态 probe。

动态 probe 不重新处理原始文本，而是直接使用离线预计算的 `feature_vec`：

```text
遍历所有 feature_dim
  ↓
读取当前候选病例在该 dim 上的值
  ↓
用 semantic_split 判断该维度能否有效切分
  ↓
选择信息增益最高的 dim
  ↓
生成追问
```

#### 5.3 第三层：逐例确认

当候选集已经很小，例如只剩 3 到 5 个病例，但没有 probe 能继续有效切分时，系统会进入逐例确认模式。

例如：

```text
“你的情况是否更像：牙齿晚上自发疼，冷热刺激后疼痛持续较久？”
```

这个阶段的目的不是再构造新的抽象维度，而是对 top candidate 进行确认

#### 6. 最终回答：answer

当系统认为可以停止时，会输出当前 top1 病例：

```text
更符合本地病例：急性牙髓炎

关键依据：
夜间自发痛、温度刺激反应、疼痛持续时间较长

处理建议：
建议尽快就诊，进行牙髓活力测试、叩诊及影像学检查；
如确诊为不可复性牙髓炎，通常需要根管治疗。
```

最终回答包含三类信息：

```text
1. 匹配到的本地病例标题；
2. 匹配依据，即用户确认过的 probe；
3. 对应病例中的处理建议 treat。
```

这里的输出应理解为“本地病例匹配结果”，而不是严格意义上的医学诊断。真正诊断仍需要医生结合口内检查、叩诊、温度测试和影像学检查。
